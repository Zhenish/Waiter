# Меню официанта — контекст для Claude Code

Сайт для приёма заказов официантами в кафе/ресторанах. Задуман как SaaS:
один и тот же сайт продаётся многим заведениям (изначально рассчитан на
100+ клиентов).

## Архитектура (важно понимать перед любыми правками)

- **Один сайт, один билд, один деплой** на GitHub Pages. НЕ собирается
  отдельный сайт под каждое кафе — так было на предыдущей итерации, потом
  сознательно отказались в пользу единого сайта (см. историю ниже).
- **Кафе (`restaurant`) определяется во время работы сайта**, а не при
  сборке. Официант вводит **PIN-код** → по нему из Supabase находится нужная
  запись в таблице `restaurants` (id, name, pin, menu — меню лежит прямо в
  JSONB-поле). После первого ввода PIN запоминается в `localStorage`
  (`src/restaurant.js`, ключ `waiter-menu-cafe-id`), дальше не спрашивается.
- **База данных одна общая** на все кафе (Supabase). Разделение — по
  `restaurant_id` в таблице `orders` и по строке в `restaurants`. Никакой
  строгой RLS-защиты сейчас нет (политики `using (true)`) — разделение
  обеспечивается фильтрацией в коде. Это осознанный компромисс для пилота на
  доверенных клиентах, задокументирован в README.
- **Меню — гибкие рубрики**, не жёстко «еда/бар». Формат:
  `{ categories: [{id, name, icon}], items: [{id, name, price, category}] }`.
  `icon` — один из фиксированного набора ключей (see `CATEGORY_ICONS` в
  `src/App.jsx`): utensils, wine, flame, cigarette, sparkles, coffee,
  package, wrench, shopping-bag, beer, ice-cream, gift.

## Ключевые файлы

- `src/App.jsx` — приложение для официантов: `PinScreen` (вход по PIN) →
  `WaiterPicker` (выбор имени официанта) → `OrderScreen` (меню, корзина,
  активные заказы, история). Роли администратора конкретного кафе здесь
  пока НЕТ — она есть только как референс, см. ниже. Стили — единый объект
  `styles`/`wrapperStyles` в конце файла (без внешнего CSS-фреймворка),
  фирменные цвета: `WINE`, `GOLD`, `INK`, `PANEL`, `PAPER`.
- `src/restaurant.js` — вся логика определения кафе для официанта:
  чтение/запись id в localStorage, поиск по PIN, поиск по сохранённому id
  (вызывается при каждом открытии — так меню всегда свежее, а отключённое
  владельцем кафе сразу блокирует вход).
- `src/supabaseClient.js` — клиент Supabase, читает `VITE_SUPABASE_URL` /
  `VITE_SUPABASE_ANON_KEY` из `.env`.
- `src/OwnerDashboard.jsx` — панель владельца сервиса: список всех кафе,
  добавление, редактирование меню (построчно и вставкой JSON целиком, режим
  дополнения — не затирает существующее), включение/отключение, удаление.
  Работает напрямую с таблицей `restaurants`.
- `src/OwnerGate.jsx` / `src/ownerAuth.js` — защита панели владельца
  паролем (`VITE_OWNER_PASSWORD`). Открывается по адресу `/#owner`
  (роутинг — в `src/main.jsx`, по `window.location.hash`, без библиотек).
- `scripts/menu-utils.mjs` — валидация формата меню (`validateMenu`),
  переиспользуется и в `push-menu.mjs` (Node), и в `OwnerDashboard.jsx`
  (браузер) — один источник правды для формата меню.
- `scripts/push-menu.mjs` — рабочий инструмент владельца сервиса для
  онбординга клиентов пачкой из терминала: читает `incoming/<id>.json`
  (`{name, categories, items}`), создаёт новую запись в `restaurants` с
  сгенерированным PIN или обновляет существующую (PIN не трогает).
  Запуск: `npm run push-menu`. (Альтернатива — то же самое можно сделать
  через панель владельца в браузере, `push-menu` удобен для пакетного
  онбординга многих кафе разом.)
- `PROMPT-FOR-AI.md` — готовый промпт для генерации файла меню через
  DeepSeek/любой ИИ в нужном формате.
- `.github/workflows/deploy.yml` — на каждый push в `main` собирает сайт
  (`npm run build`) и публикует на GitHub Pages. Секреты
  `VITE_SUPABASE_URL`/`VITE_SUPABASE_ANON_KEY` берутся из GitHub Secrets.

## Supabase — схема (см. README.md для полного SQL)

```sql
restaurants (id, name, pin, menu jsonb, created_at, updated_at)
orders (id, restaurant_id, waiter, table_number, items jsonb,
        items_count, total, created_at, completed_at)
```

## Запуск локально

```
npm install
# .env с VITE_SUPABASE_URL и VITE_SUPABASE_ANON_KEY
npm run dev
```

## Что уже встроено, а что ещё только референс

**Встроено и подключено к реальной базе:**
- `src/OwnerDashboard.jsx` + `src/OwnerGate.jsx` + `src/ownerAuth.js` —
  панель владельца сервиса, доступна на `/#owner`, защищена паролем
  (`VITE_OWNER_PASSWORD`), все действия (добавить/отредактировать
  меню/включить-отключить/удалить кафе) — настоящие запросы к таблице
  `restaurants` в Supabase, не демо-хранилище. У каждого кафе в списке есть
  кнопка «Ссылка для входа» — копирует `<сайт>/?pin=<PIN>`; официант,
  открывший её, входит автоматически без ручного набора PIN (см.
  `pinFromLinkRef` в `src/App.jsx`).
- Роль администратора КОНКРЕТНОГО кафе — вход через кнопку «Я администратор»
  на экране выбора официанта (`WaiterPicker`, `src/App.jsx`), рядом с именами
  официантов (это точка входа, а не кнопка внутри `OrderScreen` — там её
  нет). `src/AdminMenuEditor.jsx` (`AdminMenuGate`) — внешний PIN-гейт для
  входа в администрирование ВООБЩЕ, каждый раз требует повторного ввода
  PIN кафе (сверяется с `restaurant.pin`, не запоминается в localStorage).
  После входа — `src/AdminScreen.jsx`, дизайн скопирован с
  `reference-demos/team-preview.jsx`: сверху бейдж «Администратор» +
  «Сменить», под ним вкладки — «Активные» (заказы ВСЕХ официантов, только
  просмотр, без права выполнить/удалить), «Кухня/Бар» (очередь по каждой
  рубрике меню — сколько чего ждёт готовки и на каких столах), «Аналитика»
  (заказов/выручка за сегодня, средний чек, топ блюд, разбивка по
  официантам), «История», «Стоп-лист» (поиск + быстрый тумблер
  недоступности на каждом блюде — поле `item.stopped` в `restaurants.menu`;
  `OrderScreen` показывает такие блюда как недоступные, их нельзя добавить
  в заказ). Все данные заказов тянутся из `orders` по `restaurant_id`, как
  и в обычном `OrderScreen`.
  Вкладка «Меню» — ОТДЕЛЬНО защищена своим PIN-гейтом ВНУТРИ AdminScreen
  (сверяется с тем же `restaurant.pin`, но состояние разблокировки не
  переживает уход с вкладки — при возврате на «Меню» PIN спрашивается
  заново, см. `useEffect` на `tab` в `AdminScreen.jsx`). После разблокировки —
  сетка рубрик/блюд с быстрым редактированием/удалением по карточке
  (сохраняет сразу в базу через `onMenuUpdated`), плюс кнопка «Рубрики и
  вставка JSON целиком», открывающая `src/MenuEditor.jsx` — тот же полный
  редактор (добавление/удаление рубрик, вставка JSON от ИИ), что и в
  `OwnerDashboard.jsx` (используют общие `src/menuIcons.js` и
  `scripts/menu-utils.mjs`).

**Остался только как референс, НЕ встроено** — `reference-demos/team-preview.jsx`:

## История решений (коротко, чтобы не наступать на те же грабли)

- Сначала пробовали: один сайт = один официант, `window.storage`/localStorage.
- Потом: несколько официантов на одно кафе (общая история заказов, личные
  активные, роль администратора, таймер просрочки, поиск, стоп-лист).
- Потом: несколько КАФЕ — сначала через отдельную сборку сайта на каждое
  (`cafes.json` + `menus/*.json` + сборка N сайтов на GitHub Pages). Работало,
  но при 100+ клиентах правки дизайна пришлось бы применять пересборкой всех
  сайтов, а PIN тогда не был нужен вовсе (кафе разделялись путём в URL).
- Финально: перешли на текущую архитектуру (один сайт, PIN, меню в БД) —
  правки кода/дизайна применяются мгновенно у всех клиентов разом, добавление
  клиента — это просто новая строка в БД, а не пересборка.
